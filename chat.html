APRS Group Chat

A standards-based APRS messaging service built for real radios, APRS-IS clients, and club stations that want organized topic rooms without pretending packet radio is Discord.

Overview

The KM9G APRS Group Chat is a room-based messaging service that rides on normal APRS messaging. The primary transport is APRS-IS, with RF participation supported where local infrastructure allows it.

It is intentionally simple: join a room, send a message, and receive replies from other subscribed users. No special app is required beyond an APRS client that supports normal message traffic.

This service is standards-first and best-effort. If your local setup cannot get APRS-IS replies back to RF, that is usually an infrastructure issue, not a bot issue.

Core Principles

Standards First

Uses normal APRS messaging behavior instead of inventing a parallel protocol.

APRS-IS Primary

Internet-connected APRS clients get the most consistent experience. RF remains part of the picture where it makes sense.

Curated Rooms

Rooms are created by the server owner only. Users can join and leave them, but not create them.

Best-Effort Delivery

APRS is not guaranteed delivery. Delays, missed messages, and one-way behavior can happen.

Getting Started

Send APRS messages to:

KM9G-1

Start by asking for the room list or subscribing to a room you want to use.

Commands

List Rooms

ROOMS

Returns the available public rooms on the server.

Join a Room

SUB <ROOM>

Subscribes your station to a room.

Leave a Room

UNSUB <ROOM>

Removes your station from a room.

Help

HELP

Returns a short usage summary.

Sending Messages

Once you are subscribed, send a normal APRS message to the chat node. Your message is forwarded to other subscribers in that room.

SUB GENERAL
Hello everyone

Rooms are intentionally curated and topic-based. Example room names may include GENERAL, TECH, LOCAL, and EVENTS.

Room Policy

Rooms are created and managed by the server owner. Public users cannot create rooms.

  • Known rooms can be joined with SUB <ROOM>.
  • Unknown room names are rejected.
  • This keeps the system readable, intentional, and efficient over APRS.

Message Limits

Messages must be 67 characters or less.

Messages that exceed the limit are rejected and not forwarded. The sender receives a reply like:

Msg too long (83 chars). Max 67. Try again.

This keeps behavior predictable across a wide range of APRS gear and clients.

Delivery Expectations

Internet-Connected Clients

These generally have the most reliable experience because they talk directly to APRS-IS.

RF Users Behind Bidirectional iGates

These users can usually participate normally when local infrastructure is healthy.

RF Users Behind RX-Only iGates

These users may be able to send messages in but not receive replies back. That is a network limitation.

If you want better APRS coverage and better message return paths, the answer is simple: build better infrastructure.

Public Dashboard

A public dashboard shows system status, room activity, and recent message snapshots. It is intentionally published as a static site rather than a live public-facing server.

  • Updated periodically, not in real time
  • Built for low overhead and safe operation from a mobile RV-based station
  • Can remain stale while the station is offline or moving

Status and Offline Behavior

The dashboard shows a last-updated time and uses a browser-side failsafe. If no fresh update is seen for more than 10 minutes, it automatically switches to an offline state.

Group chat is Temporarily Offline... get it?

This makes the public page honest even when the station has lost connectivity or is deliberately shut down.

For Clubs

Clubs can run their own deployment using the same software stack. On-air use remains free. The value is in the software, packaging, dashboard, and support around it.

  • Curated room sets for club use
  • Club-owned nodes and dashboards
  • Optional local RF optimization
  • Future expansion for additional services
  • Software is available to patreon supporters or youtube channel members (any teir) upon request

Philosophy

This project leans into what APRS already is: open, distributed, infrastructure-driven, and occasionally gloriously imperfect.

It does not try to paper over every network weakness with hacks. It uses the protocol correctly, tells the truth about delivery, and encourages better infrastructure where it matters.

Built for APRS users who would rather improve the network than complain about it.
